## Critical Issues

### 🚨 Issue 1: Authentication Documentation Conflict

**Severity**: HIGH - Critical documentation discrepancy

**Problem**:
Two different documentation locations describe completely different authentication systems:

1. **TaskDocs-BlockSecOps/phases/api/API-Authentication.md**:
   - Documents HttpOnly cookies
   - JWT tokens generated by API service itself
   - bcrypt + SHA-256 password hashing
   - 30-minute access tokens stored in cookies

2. **blocksecops-docs/api/endpoints-reference.md** & **architecture/authentication-system.md**:
   - Documents Supabase Auth integration
   - ES256 JWT tokens from Supabase
   - OAuth providers (Google, Microsoft, GitHub)
   - Token verification via JWKS

**Current Reality** (verified in source code):
- API service uses Supabase Auth (`src/infrastructure/auth/supabase_client.py`)
- JWT verification with ES256 algorithm
- Supabase handles all auth flows

**Impact**:
- TaskDocs API documentation is **OUTDATED**
- Developers following TaskDocs will implement wrong authentication
- Major confusion between legacy and current systems

**Recommendation**:
1. **URGENT**: Update TaskDocs-BlockSecOps/phases/api/API-Authentication.md to reflect Supabase Auth
2. Archive old HttpOnly cookie documentation as "Legacy Authentication (v0.1-v0.3)"
3. Add clear version markers showing when the auth system changed
4. Cross-reference between TaskDocs and blocksecops-docs
